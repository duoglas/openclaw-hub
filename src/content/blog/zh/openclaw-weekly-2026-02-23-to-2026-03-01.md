如果只用一句话总结这周：

**OpenClaw 进入“能跑”到“要稳”的阶段。**

模型能力不是本周重点，重点是三件更现实的事：

1. 消息投递可靠性（静默丢失 / 重复投递）
2. Telegram 高发故障（尤其 409 Conflict）
3. 安全边界与默认策略收紧（配置不对就容易踩坑）

---

## 本周最值得关注的 3 条主线

## 1）消息可靠性成了头号痛点

近 7 天社区反馈非常集中：

- 机器人“看起来在线”，但消息被静默丢弃
- 崩溃恢复后出现重复投递
- 缺少可观测性，定位链路成本很高

这不是单点 bug，而是“投递链路治理”问题。

**对业务的真实影响：**
- 自动化流程出现“看不见的失败”
- 运营误判“任务已完成”
- 排障时间被显著拉长

## 2）Telegram 故障继续高发，409 Conflict 是典型问题

这周最典型的错误之一：

`409 Conflict: terminated by other getUpdates request`

本质原因通常不是“网络差”，而是 **同一 bot token 被多个实例同时轮询**。

常见触发场景：
- 本地手工起过一份、systemd 里又跑一份
- Docker 多副本同时消费同一个 token
- webhook 残留 + polling 并存

本周我们已经发布了对应 checklist（中英双语）：
- [OpenClaw Telegram 409 Conflict 一次性排查（2026）](/zh/blog/openclaw-telegram-409-conflict-getupdates-fix-2026/)

## 3）安全边界持续收紧：默认配置不再“随便跑”

本周趋势里，另一个明确信号是：

- origins / allowFrom / safeBins / SSRF 等边界越来越严格
- 过去“能跑”的宽松配置，未来可能变成风险点

对个人用户和小团队的启发很直接：

**别把“跑起来”当完成，必须补“最小权限 + 可观测 + 单实例约束”。**

## 下周可直接执行的动作（不是口号）

### 动作 A：做一次“单实例体检”（10 分钟）

目标：确保没有重复 polling。

```bash
openclaw status
ps -ef | grep -E "openclaw|node" | grep -v grep
systemctl --user status openclaw-gateway
```

### 动作 B：清理 Telegram webhook 残留

目标：避免 webhook 与 polling 冲突。

```bash
# 查看 webhook 状态
curl -s "https://api.telegram.org/bot<YOUR_BOT_TOKEN>/getWebhookInfo"

# 如需切回 polling，删除 webhook
curl -s "https://api.telegram.org/bot<YOUR_BOT_TOKEN>/deleteWebhook"
```

### 动作 C：给关键链路补最小观测

至少记录三类日志字段：
- 入站消息时间戳
- 投递目标（channel / thread / topic）
- 投递结果（成功/失败+错误码）

没有这三项，后面排障只能靠猜。
